这是工作流可执行性的核心缺口。

问题性质
当前 tasks.md 是"人类可读的任务清单"，但不是"agent 可执行的任务规范"。这导致：

1. 缺少执行约束
status：agent 无法跟踪任务状态，不知道哪些任务已完成、哪些进行中
verification command：agent 无法自主验证任务完成，需要人工判断
done when：缺少明确的完成标准，容易出现"做了但没做完"的情况
owner：责任归属不明确
2. 影响范围
P0 流程可以执行，但存在这些问题：

Agent 需要人工判断"这个任务做完了吗"
无法自动化验证任务质量
任务推进依赖 agent 的主观判断，不够稳定
小功能链路的测试任务会引用不存在的 stories.md
3. 本质差异

当前状态：文档驱动开发
- tasks.md 是"指导手册"
- agent 读文档 → 理解 → 实现 → 人工判断是否完成

目标状态：可执行工作流
- tasks.md 是"执行规范"
- agent 读任务 → 执行 → 运行验证命令 → 自动判断完成
解决方向
需要将 tasks.md 升级为"可执行任务规范"：

添加任务元数据：status、verification、done when、owner
适配小功能链路：stories.md 改为可选引用
提供验证命令：每个任务都有明确的验证方式
这样 agent 就能：

自主跟踪任务状态
自动验证任务完成
稳定推进任务执行
这是从 P0（流程可用）到 P1（流程高效）的关键升级。
