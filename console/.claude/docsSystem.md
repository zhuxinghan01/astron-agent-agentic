SaaS 文档系统（轻文档 + Diff 驱动迁移版）

1. 背景

Astron Agent 已进入多渠道并行迭代阶段。开源版、SaaS 版、私有化版、海外版会独立演进，但研发方法应保持同源。

SaaS 侧当前的核心问题不是缺少模板，而是每次 Merge Request 合并后，没有稳定留下可检索、可追溯、可筛选的结构化记录。这会直接导致：

- 变更历史难检索，复盘成本高
- Reviewer 只能看当下 MR，知识无法沉淀
- 很难筛出哪些 SaaS 变更值得开源版关注
- 多渠道分叉后，决策和经验容易丢失

第一阶段目标只有一个：

所有 MR 合并后，必须留下可检索的文档记录。

2. 系统定位

SaaS 文档系统不是一套重文档体系，也不是把代码实现细节重新写一遍。

它的职责是：

- 记录这次改动做了什么
- 沉淀为什么这么做
- 说明哪些约束必须保留
- 保留对开源版是否需要关注的判断

它不承担的职责是：

- 不要求把代码细节完整翻译成文档
- 不要求 SaaS 与开源版代码同步
- 不要求现阶段引入 CI 门禁
- 不在仓库内维护一套额外的 Git merge-log 目录

一句话原则：

文档负责设计、决策、约束；MR diff 负责实现事实。

3. 建设目标

SaaS 文档系统围绕 MR merge 事件运转，目标如下：

1. 每次 MR 都有最小记录，知道“改了什么”
2. 重要变更可以继续上卷，知道“目标、边界、影响是什么”
3. 架构和关键策略变化有正式决策记录，知道“为什么这么改”
4. 能按统一字段筛出需要开源版关注的内容
5. 先依赖团队约定和 Review 习惯运行，不增加 CI 校验门禁

4. 设计原则

4.1 方法同源，不要求代码同源

各渠道未来可独立演进，统一的是研发方法、信息结构和决策留痕，不是代码实现本身。

4.2 以 MR 为入口，而不是以写文档为入口

研发动作天然围绕 GitLab MR 发生，因此文档系统必须绑定 MR 流程，让补文档成为 merge 的一部分。

4.3 轻量起步，逐层升级

不是每个 MR 都值得写长文档。所有变更先留下 L0，只有重要变更再升级到 L1 或 L2。

4.4 结构化优先，叙述性次之

优先保证字段可筛选、状态可追踪、责任可明确；长文档只在必要时补充。

4.5 文档保留高价值信息，代码细节以 Diff 为准

L1/L2 文档不要求完整复述 DTO、Entity、API、类结构、重构路径等实现细节。相关事实以 MR diff 为准。

文档重点记录：

- 背景与目标
- 关键决策
- 业务约束
- 迁移边界
- 开源适配建议
- 验收标准

5. 文档载体

SaaS 文档系统采用飞书作为主要载体，不建设 Git 文档仓库。

5.1 载体划分

- 飞书多维表格：统一入口，承载所有 MR 的结构化记录
- 飞书详细文档：承载 L1 级别的重要变更说明
- ADR 文档：承载 L2 级别的架构或关键策略决策
- GitLab MR：作为研发上下文和代码入口，负责与飞书双向关联

5.2 不做的事情

- 不在 SaaS 仓库额外维护 merge-log 目录
- 不引入 skills 自动化链路
- 现阶段不增加 CI 校验门禁

7. 升级规则

7.1 升级到 L1

满足以下任一条件时，MR 需要从 L0 升级到 L1：

- 新增或删除 API 端点
- 修改数据模型、Entity 或表结构
- 变更用户可见行为
- 修复 High 及以上级别 Bug

7.2 升级到 L2

满足以下任一条件时，MR 需要继续升级到 L2：

- 新增或替换技术组件
- 修改认证或权限模型
- 跨服务架构调整
- 变更部署拓扑
- 关键技术策略调整

7.3 关系说明

- L2 不是替代 L0
- L2 通常意味着：L0 表格记录 + 必要时 L1 说明 + L2 ADR
- 所有高级文档都必须回链到原始 MR 记录

开发者创建 MR 时，至少填写以下内容：

<!-- 一句话说清楚改了什么 -->

<!-- 涉及哪些模块/服务 -->

这部分信息直接进入飞书多维表格，是 L0 的最小输入。
10. 标准流程

6. 开发者创建 GitLab MR，并按模板填写描述
7. GitLab Webhook 将 MR 基础信息同步到飞书多维表格
8. Reviewer 在 Review 过程中判断文档层级
9. 如果停留在 L0，表格记录完整即可
10. 如果升级到 L1，提交人补飞书详细文档，并把链接回填到表格
11. 如果升级到 L2，技术负责人补 ADR，并把链接回填到表格
12. MR 描述中贴飞书链接，飞书表格中保留 GitLab MR 链接，形成双向关联

10.2 执行要求

- 没有 L0 记录的 MR，不算完成文档补录
- 被判断为 L1/L2 的 MR，必须补齐对应文档链接
- 现阶段靠团队约定和 Code Review 执行，不通过 CI 卡口实现

11. L1 文档要求

当 MR 被判断为 L1 时，需要补一份飞书详细文档。L1 的目标不是写成长篇实现说明，而是让后来者和开源迁移方快速理解这次改动的意图、边界和约束。

15. 关键业务规则
16. 数据或接口是否有破坏性变化
17. SaaS 特有依赖

19. 非目标范围

11.2 写作要求

- 不要求完整罗列 DTO、Entity、API 字段和类结构
- 具体实现、字段变化、重构路径以 MR diff 为准
- 重点写清哪些能力必须保持一致，哪些属于 SaaS 特有能力，哪些可以裁剪

11.3 L1 需要回答的问题

- 这次改动解决什么问题
- 这次交付的核心能力是什么
- 哪些业务规则不能丢
- 哪些部分是 SaaS 专属，不应直接迁到开源版
- 如果未来迁移到开源版，建议保留什么、删减什么
- 验收时应重点检查什么

12. L2 ADR 要求

当 MR 被判断为 L2 时，需要补 ADR 文档。L2 的目标不是记录实现细节，而是沉淀关键判断和不可轻易打破的约束。

- 为什么要做这个决策

- 哪些设计约束后续不能被随意破坏
- 对 SaaS 和开源版分别有什么影响

13. 文档与 Diff 的职责边界

这是 SaaS 文档系统的核心约束。

13.1 文档负责什么

- 背景和目标
- 设计思路
- 关键决策
- 业务约束
- 迁移边界
- 开源适配建议
- 验收标准

13.2 Diff 负责什么

- 改了哪些文件
- 新增了哪些接口、字段、实体、表
- 具体逻辑怎么实现
- 实际的重构路径
- 代码层面的实现细节

13.3 制度性表述

可直接写入团队约定：

L1/L2 文档不要求完整复述代码实现，具体实现以 MR diff 为准；文档重点记录设计意图、关键决策、业务约束和开源迁移建议。

14. 面向开源迁移的使用方式

未来需要将 SaaS 能力迁移到开源版时，优先使用以下输入，而不是要求团队先补一篇超细实现文档：

- MR URL 或 MR diff
- L1/L2 文档
- 开源仓库当前相关模块代码
- 明确的迁移任务描述

建议的任务表达方式：

把 SaaS 这次能力迁到开源版，明确哪些行为必须一致，哪些 SaaS 专属能力可以不迁，按当前开源代码结构给出迁移方案和实现草稿。

这样做的目的：

- 让文档保持轻量，降低维护成本
- 让 AI 直接读取实现事实，而不是读取一份已经过时的“实现翻译稿”
- 把团队精力集中在高价值信息上，即目标、约束、边界和验收

15. 开源版衔接机制

SaaS 文档系统不要求与开源版同步开发，但要求保留“哪些内容值得开源关注”的判断。

15.1 判断方式

通过 是否开源 字段进行初筛：

- yes：明确值得开源版关注
- maybe：可能有关，需要进一步评估
- no：明确无关
- unknown：暂未判断，后续补充

15.2 跟进方式

通过 开源进度 字段跟踪后续动作：

- pending：待评估或待开源跟进
- in_progress：已进入处理
- done：已完成处理或同步
- not_needed：经评估无需跟进

15.3 使用方式

每个迭代结束时，筛选：

- 是否开源 in (yes, maybe)
- 且 开源进度 in (pending, in_progress)

这些记录作为开源版规划输入，而不是自动触发同步。

16. 历史数据处理

为保证系统可用，需要补齐一部分历史 MR 数据。

建议做法：

29. 通过 GitLab API 批量拉取历史 MR
30. 先导入飞书多维表格
31. 是否开源 默认标记为 unknown
32. 由团队成员按模块认领，逐步补充判断

历史数据不要求一次性清理完成，但必须保证后续新 MR 全量进入系统。

17. 当前阶段运行规则

- 强制要求：所有 merge 必须补 L0 记录
- 升级要求：命中规则的 MR 必须升级到 L1 或 L2
- 文档要求：L1/L2 重点记录意图、约束和迁移建议，不复述实现细节
- 实现来源：具体实现以 MR diff 为准
- 执行方式：靠团队约定和 Reviewer 把关
- 治理方式：按迭代周期筛选需要开源关注的记录
- 演进方向：后续再评估是否引入自动校验或更强约束

18. 总结

SaaS 文档系统的本质，不是把 MR 变成一篇长文档，而是把“MR 追踪”升级为“分层知识沉淀”。

当前最小闭环是：

MR 创建 -> 表格留痕 -> Reviewer 判断层级 -> 必要时补 L1/L2 -> 周期性筛选开源跟进项

这套机制先解决三个问题：

- 每次变更都有记录
- 重要变更有高价值说明，而不是重复实现细节
- SaaS 演进过程中，能持续沉淀可复用的决策、约束和迁移判断

一句话结论：

少写实现清单，多写意图、约束和决策；迁移时让 AI 读 diff，而不是让人手工把 diff 重写一遍。
